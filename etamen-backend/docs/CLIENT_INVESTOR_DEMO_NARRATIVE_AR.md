# سرد عرض العميل والمستثمر

هذا المستند مخصص لشرح Etamen في عرض محلي داخلي أو عرض تمهيدي مغلق. لا يعني أن المنتج جاهز للإطلاق أو production أو staging.

## المشكلة

تجربة الصحة الرقمية غالبا متفرقة: حجز الدكتور في مكان، نتائج الأشعة في مكان، الدفع والمتابعة في مكان آخر، وإدارة المزودين والدعم لا تكون متصلة بنفس التجربة.

هذا يخلق مشاكل واضحة:

- المريض يضيع بين خدمات متعددة.
- المزود لا يرى صورة تشغيل موحدة.
- المنصة تحتاج مراجعة مدفوعات، دعم، ونزاعات بدون تسريب بيانات.
- أي منتج صحي يحتاج ضوابط سلامة وخصوصية قبل التعامل مع مستخدمين حقيقيين.

## حل Etamen

Etamen يقدم نموذج health super-app محلي يجمع:

- تجربة مريض للحجز والمتابعة ورفع إثبات الدفع.
- workspaces للمزودين لإدارة العمليات حسب الصلاحيات.
- Operations Center للأدمن لمراجعة المدفوعات والمزودين والدعم والاسترداد والنزاعات.
- SOPs وguardrails لعدم تشخيص المستخدم أو كشف بيانات خاصة.

## لماذا super-app؟

لأن رحلة الصحة ليست خدمة واحدة. المريض قد يبدأ بحجز دكتور، يحتاج أشعة أو تحليل، ثم يتابع دعم أو استرداد. وجود هذه الرحلة في إطار واحد يجعل المنتج قابلا للتوسع التشغيلي، بشرط أن يظل backend هو مصدر الحقيقة للأسعار والحالات والصلاحيات.

## قيمة المريض

- خدمات متعددة من مكان واحد.
- حجز دكتور ومتابعة المواعيد.
- Radiology/Gym/Coach flows محليا.
- رفع إثبات دفع يدوي ومراجعة واضحة.
- دعم، استرداد، ونزاعات داخل التطبيق.
- رسائل سلامة تمنع التشخيص أو التعامل مع الطوارئ كبديل للطبيب.

## قيمة المزود

- workspace داخل نفس التطبيق.
- dashboard وoperations MVP.
- صلاحيات staff من backend.
- رؤية حجوزات أو طلبات حسب نوع المزود.
- دعم مرتبط بالمزود بدون كشف ملاحظات داخلية خاصة بالأدمن.

## قيمة المنصة والأدمن

- مراجعة مدفوعات proof-based.
- طابور موافقات مزودين.
- دعم واسترداد ونزاعات.
- Audit log آمن.
- منع غير الأدمن من admin APIs.
- منع staff من مزود آخر.

## احتمالات الإيرادات

هذه احتمالات مستقبلية وليست إيرادا نشطا حاليا:

- عمولة على الحجوزات المؤكدة.
- اشتراكات مزودين مستقبلية.
- خدمات تشغيلية premium للمزودين.
- باقات مؤسسية للمستشفيات أو المراكز بعد موافقات قانونية وتشغيلية.

لا يتم الادعاء بوجود revenue live أو live payment حاليا.

## الحالة الحالية

مقبول محليا فقط:

- patient flows.
- provider workspaces and operations MVP.
- platform admin operations.
- support/refund/dispute foundation.
- local real-phone gate سابقا.
- internal demo package.

## ما ليس جاهزا

- staging غير معتمد.
- production غير معتمد.
- app-store غير معتمد.
- لا يوجد مستخدمون حقيقيون.
- لا يوجد live Paymob أو live refund gateway.
- لا توجد موافقات قانونية نهائية.
- لا يوجد load testing أو backup/restore server validation.

## البوابة التقنية التالية

استعادة server access ثم تنفيذ staging deployment آمن بعد backup، وتشغيل readiness/data checks وreal-phone staging gate قبل أي supervised pilot.
